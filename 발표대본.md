## 슬라이드 1 — Title (약 30초)

안녕하세요. 오늘 발표할 프로젝트는 **NEON DRIFT** 입니다.

호버크래프트를 직접 조종해서 기지를 방어하는 네온 스타일 아케이드 디펜스 게임이고요,
**언리얼 엔진 5 위에서 C++로만** 처음부터 끝까지 직접 만든 1인 프로젝트입니다.

블루프린트로 게임 로직을 짠 부분이 단 한 군데도 없다는 게 이번 발표의 핵심 포인트 중 하나입니다.

## 슬라이드 2 — Concept (약 1분)

먼저 게임이 어떤 모습인지 간단히 소개드리겠습니다.

플레이어는 호버크래프트를 타고 맵을 자유롭게 비행하면서 자원 블록을 부숩니다.
부서진 블록에서 흩어지는 자원 부스러기를 **자석으로 흡수**해서 모으고요,
정해진 시간이 지나면 사방에서 몬스터가 밀려오는 **웨이브 페이즈**가 시작됩니다.

이때 플레이어는 두 가지 선택지가 있습니다.
**호버크래프트에 탄 채로 직접 응전**하거나, 아니면 **기지 중앙의 수동 터렛에 탑승**해서 화력을 집중하는 거죠.
기지 주변에는 자동 터렛도 배치돼 있어서, 이걸 업그레이드로 늘려나가는 것도 전략의 한 축입니다.

이 루프를 **5웨이브 동안 버텨내는 것**이 목표고요,
장르는 액션 리소스 디펜스, **3일 스프린트 기간으로 1인 개발**한 PC용 프로토타입입니다.

## 슬라이드 3 — Core Loop (약 45초)

코어 루프는 네 단계로 나뉩니다.

**비행 → 자원 채취 → 웨이브 방어 → 업그레이드.**
그리고 다시 비행으로 돌아갑니다.

채취 단계에서 모은 자원이 곧 전투력이 되고, 전투 단계에서 살아남아야 다음 업그레이드로 이어집니다.
이 네 단계가 **한 사이클을 이루면서 5웨이브 끝까지 가속**되는 구조입니다.

특히 마지막 업그레이드 단계는 **기체 4종 · 터렛 4종, 총 8가지 스탯**을 자유롭게 선택할 수 있어서,
플레이어마다 다른 빌드로 게임을 끌어갈 수 있게 했습니다.

이 루프가 실제로 어떻게 돌아가는지, 영상으로 직접 보여드리겠습니다.

## 슬라이드 4 — Live Demo (약 5분) [ 시연 영상 ]

> 영상이 재생되는 동안 화면을 함께 보면서 설명. 시간은 절대값이 아니라 **웨이브 진행 기준**으로 맞춰서 말하기.

### ▸ 영상 시작 ~ 첫 비행 (Pre-Wave)

지금 보시는 게 게임 시작 직후 모습입니다.

조작감부터 잠깐 짚자면, 이 호버크래프트는 **중력이 없는 6DOF 비행체**입니다.
일반 캐릭터 컨트롤러를 쓴 게 아니라, 매 틱마다 **속도 벡터를 직접 적분**하고
거기에 **지수적 드래그**를 곱해서 자연스럽게 감속하도록 직접 짰습니다.
관성이 살아있는 미끄러지는 느낌이 그래서 나오는 거고요.

지금 카메라 화각이 살짝 넓어졌는데, **속도에 따라 FOV가 70도에서 110도까지 동적으로 보간**됩니다.
가속할수록 시야가 벌어져서 속도감을 시각적으로 강조하게끔 했습니다.

그리고 입력 시스템이 좀 특별합니다. 언리얼 5의 **Enhanced Input** 을 쓰긴 했는데,
보통은 에디터에서 `.uasset` 으로 InputMappingContext 같은 걸 만들거든요.
저는 그걸 다 **PlayerController가 런타임에 C++로 NewObject 생성**해서 매핑합니다.
**입력 에셋이 단 하나도 없습니다.** 전부 코드입니다.

### ▸ 자원 채취 (Gather Phase)

지금 보이는 빛나는 큐브들이 **자원 블록**입니다.
색깔별로 세 종류가 있는데 — **금(주황), 철(시안), 다이아(흰색)** — 각자 다른 **공격력 요구치**가 있습니다.

방금 다이아 블록을 쐈는데 스파크만 튀고 안 부서졌죠.
이건 `IDamageable` 인터페이스에 **AttackerPower 매개변수**를 같이 넘기는 구조 덕분입니다.
블록이 자기 RequiredPower와 비교해서, 못 미치면 데미지 자체가 0으로 처리됩니다.
**플레이어에게 자연스럽게 업그레이드를 요구하는 게이트**가 되는 거죠.

블록을 부수면 부스러기가 흩어지고, 호버크래프트 근처로 다가가면 **빨려 들어옵니다.**
이게 자석 시스템인데, `TActorIterator` 로 매 틱마다 일정 반경 안의 Shard를 찾아서
**가속 보간**으로 끌어당기는 식입니다. 자석 반경 자체도 업그레이드 항목이에요.

### ▸ 첫 웨이브 시작 (Combat Phase)

이제 F키로 채취 단계를 조기 종료하면, 사방에서 몬스터가 밀려오기 시작합니다.

게임 전체가 **7단계 상태머신**으로 돌아갑니다 — MainMenu, PreWave, Gather, Combat, Shop, Victory, GameOver.
**GameMode가 페이즈를 소유**하고, 다른 모든 액터는 GameMode를 참조해서 자기 상태를 결정합니다.
플레이어가 발사가 안 되거나, 몬스터가 안 움직이는 등의 페이즈별 동작은 전부 여기서 갈리고요,
이게 **단일 진실 공급원(single source of truth)** 으로 작동해서 디버깅이 굉장히 깔끔합니다.

### ▸ 터렛 탑승

지금 E키 누르는 거 보이시나요? 기지 중앙 수동 터렛에 탑승합니다.

이때 **카메라가 부드럽게 블렌딩**되면서 시점이 옮겨가는데, 이건 PlayerController의 `SetViewTargetWithBlend` 를 활용했습니다.
탑승 중에는 **호버크래프트는 멈춰있고** (속도 0으로 잠금), 입력은 그대로 터렛 조준에 라우팅됩니다.

화면 중앙에 보이는 흰 십자선과 작은 노란 점, 이게 **이중 크로스헤어**입니다.
큰 흰색은 카메라 중심 — 즉 플레이어가 보는 방향이고,
노란 점은 **실제 포신이 향하는 곳**을 `ProjectWorldLocationToScreen` 으로 화면에 투영한 겁니다.
포신은 회전 속도 제한이 있어서 카메라 움직임을 즉시 따라가지 못하거든요.
**둘 사이의 간극이 곧 조준 지연 피드백**으로 작동해서, 무게감 있는 사격 느낌을 줍니다.

### ▸ 웨이브 클리어 → 상점

웨이브1 클리어. 5마리 잡으면 끝납니다.

자동으로 **Shop 페이즈**로 전환되는데, 지금 보시는 UI가 **순수 C++ UMG** 입니다.
`UUserWidget` 을 상속한 위젯을 코드에서 직접 생성하고,
`UCanvasPanelSlot`을 통해서 모든 자식을 좌표로 배치합니다.
**디자이너 툴에서 끌어다 놓은 위젯이 없습니다.** 전부 코드로 짠 UI예요.

업그레이드는 **GameInstance에 영속화**됩니다.
GameInstance가 살아있는 한 — 즉 재시작 전까지 — 강화 상태가 유지되고,
**파생 스탯 계산도 모두 GameInstance가 책임집니다.** 기체와 터렛이 GameInstance에서 스탯을 받아 적용하는 구조입니다.

### ▸ 웨이브 2~3 — 다입구 진입

웨이브가 올라가면서 **몬스터 진입 방향이 추가**됩니다.
2웨이브는 북쪽 한 방향이지만, 3웨이브부터는 북·동에서 동시에 들어와요.

스폰 위치는 레벨에 배치한 `ASpawnPoint` 액터들이고,
GameMode가 `TActorIterator` 로 BeginPlay 때 한 번 수집해서, 웨이브 정의에 따라 활성화합니다.
**각 입구마다 독립적인 스폰 펌프**가 돌아서, 동시에 여러 방향으로 압박이 들어오는 구조입니다.

### ▸ 웨이브 4~5 — 전방위

자, 4웨이브부터는 3방향, 5웨이브는 **N·E·S·W 사방에서 동시 스폰**입니다.

이때부터는 자동 터렛이 핵심이에요.
업그레이드 항목 중에 **AutoTurret_Count** 가 있는데, 단계가 오를 때마다 자동 터렛이 한 대씩 활성화됩니다.
GameMode가 ApplyUpgrade에서 `bEnabled` 플래그를 켜주는 식이고,
각 자동 터렛은 **자체적으로 최근접 몬스터를 재탐색**하면서 자율적으로 사격합니다.

### ▸ 5웨이브 클리어 → Victory

마지막 30마리까지 잡으면 Victory 화면입니다.

상태머신이 `Victory` 로 전이하면서, 모든 액터의 동작이 정지되고 결과 화면이 그려집니다.
R키로 재시작하면 GameInstance가 `ResetRun()` 을 호출해서 강화 상태까지 깨끗하게 초기화하고,
다시 MainMenu로 돌아가는 루프입니다.

## 슬라이드 5 — Tech Stack (약 1분 30초)

영상으로 본 시스템들을 기술 스택으로 정리하면 이렇습니다.

먼저 **엔진은 언리얼 5**, **언어는 C++만** 사용했습니다.
앞서 말씀드렸듯이 **블루프린트 게임 로직 비율은 0%** 입니다.

특히 신경 쓴 부분 몇 가지만 짚어드리면 —

**입력**은 Enhanced Input을 쓰되 `.uasset` 없이 전부 C++로 런타임 생성했고요,
**UI**는 UMG Canvas를 쓰되 위젯 트리를 모두 C++로 직접 만들었습니다.
**이펙트**는 Niagara 트레일에 Dynamic Material Instance를 연결해서, 피격 플래시나 네온 펄스 같은 색 변화를 코드로 제어합니다.
**오디오**는 AudioComponent의 피치와 볼륨을 호버크래프트 속도와 연동시켜서, 가속할 때 자연스럽게 엔진 사운드가 고조되도록 했습니다.

오른쪽 목록은 **직접 구현한 커스텀 시스템 10가지** 인데, 영상에서 한 번씩 다 짚었던 것들이에요.
이 중에서 발표 시간상 깊게 못 다룬 게 **업그레이드 영속화**랑 **UMG를 C++로 구축한 부분**입니다.

업그레이드 영속화는 `UNeonGameInstance` 가 강화 상태를 들고 있어서, 웨이브가 바뀌어도, 캐릭터를 다시 스폰해도 강화가 유지됩니다.
파생 스탯 계산도 전부 GameInstance가 단일 진실 공급원으로 책임지는 구조고요.

UMG의 C++ 구축은, 사실 보통은 이걸 다 에디터의 위젯 블루프린트로 만들거든요.
저는 이걸 `UCanvasPanelSlot`을 코드에서 직접 조작해서 위치·크기·표시 여부를 모두 제어하는 방식으로 짰습니다.
그래서 **위젯 블루프린트 에셋이 하나도 없습니다.** 빌드하면 코드 변경만으로 UI 변경이 추적되니까, 버전 관리가 깔끔해지는 장점이 있습니다.

## 슬라이드 6 — Architecture (약 1분)

클래스 구조는 크게 세 덩어리입니다.

**Game Flow 쪽** — GameMode, GameInstance, PlayerController.
이 셋이 게임 전체의 흐름과 영속 상태, 입력 라우팅을 담당합니다.

**Gameplay Actors 쪽** — PlayerShip, ManualTurret, AutoTurret, Base, ResourceBlock, Monster.
공격을 받는 액터들은 전부 **`IDamageable` 인터페이스**를 구현해서, 발사 라인트레이스가 누구를 맞히든 동일한 `TakeHit` 호출로 처리됩니다.
타입 캐스팅이 사라지고 확장도 쉬워지는 클래식한 패턴이죠.

**UI 쪽** — AHUD와 UUserWidget.
크로스헤어처럼 매 프레임 좌표 계산이 필요한 건 AHUD의 Canvas로,
페이즈별로 들고 나는 UI는 UMG Widget으로 분리했습니다.

전체 클래스 수가 많지 않은데, 이게 의도된 부분입니다.
**가능한 만큼 책임을 한 곳에 모았고**, GameMode가 흐름을, GameInstance가 영속 상태를, PlayerController가 입력을 — 이렇게 SRP를 지키려고 했습니다.

## 슬라이드 7 — State Machine (약 45초)

마지막으로, 게임 루프 상태머신을 한 장에 정리한 다이어그램입니다.

**MainMenu → PreWave → Gather → Combat → Shop → 다시 PreWave** 로 루프하고,
5웨이브를 다 클리어하면 **Victory**, 기지 HP가 0이 되면 **GameOver** 로 빠집니다.

아까도 잠깐 말씀드렸지만, **이 페이즈 변수는 오직 GameMode 한 곳에만 존재**합니다.
모든 다른 액터 — 플레이어, 몬스터, 터렛, UI — 가 자기 동작을 결정할 때 `GameMode->Phase` 를 참조하지,
**중복된 상태 변수를 들고 있지 않습니다.**

이 원칙 하나만 지켜도 **상태가 갈라지는 버그를 구조적으로 차단**할 수 있어서,
프로토타입 규모에서는 압도적으로 빠르게 짤 수 있었습니다.

## 💻 코드 관련 예상 질문

### 1. "Enhanced Input을 `.uasset` 없이 코드로만 어떻게 구성하셨나요?"

`PlayerController::BeginPlay()`에서 IMC/IA를 코드로 동적 생성합니다.

```cpp
void ANeonPlayerController::BeginPlay()
{
    Super::BeginPlay();

    IMC = NewObject<UInputMappingContext>(this);
    IA_Move = NewObject<UInputAction>(this);
    IA_Move->ValueType = EInputActionValueType::Axis2D;

    IMC->MapKey(IA_Move, EKeys::W);
    // S, A, D는 SwizzleAxis / Negate Modifier로 축 변환
    // ...

    auto* Subsystem = GetLocalPlayer()->GetSubsystem<UEnhancedInputLocalPlayerSubsystem>();
    Subsystem->AddMappingContext(IMC, /*Priority=*/0);
}
```

그다음 Pawn의 `SetupPlayerInputComponent`에서 `UEnhancedInputComponent::BindAction(IA, ETriggerEvent::Triggered, ...)`으로 핸들러를 연결합니다.

UE 정식 지원 경로라 호환성 이슈는 없고, **입력 에셋 0개**로 모든 매핑이 코드에 들어가서 키 변경이 Git 디프로 추적됩니다. 입력이 자주 바뀌는 프로토타입 단계와 정합성이 좋은 방식이에요.

### 2. "호버크래프트 비행 물리는 구체적으로 어떤 식으로 짜셨나요?"

`CharacterMovementComponent` 안 쓰고 매 틱 Velocity를 직접 적분합니다.

```cpp
void APlayerShip::Tick(float Dt)
{
    Velocity += Acceleration * InputDir * Dt;              // 입력 → 가속
    Velocity *= FMath::Pow(DragCoeff, Dt);                 // 시간 기반 지수 감쇠
    Velocity = Velocity.GetClampedToMaxSize(MaxSpeed);
    AddActorWorldOffset(Velocity * Dt, /*sweep=*/true);    // 충돌 sweep

    const float t = Velocity.Size() / MaxSpeed;
    Camera->SetFieldOfView(FMath::Lerp(70.f, 110.f, t));   // 속도 비례 FOV
}
```

핵심은 감쇠를 `Velocity *= 0.95f`가 아니라 `Velocity *= FMath::Pow(DragCoeff, Dt)`로 쓴 부분이에요. 전자는 프레임 의존이라 60fps와 120fps에서 감쇠 속도가 두 배 차이 나지만, 후자는 시간 기준이라 항상 일정합니다.

중력은 0이고 Z축에만 약한 추가 감쇠를 줘서 입력을 떼면 자연스럽게 호버 높이로 수렴하도록 했습니다. FOV 보간은 가속감을 시각적으로 강조하는 흔한 기법이고요.

### 3. "`IDamageable` 을 추상 클래스 대신 인터페이스로 한 이유가 있나요?"

UE의 UClass가 **단일 상속**이라서요. `ABase`, `AMonster`, `AResourceBlock`이 이미 `AActor`(또는 `APawn`)를 상속받고 있어서 추상 베이스 클래스를 추가로 둘 수 없습니다. 인터페이스가 사실상 강제 선택이고요.

UE 인터페이스는 1급 시민이라 다음 패턴으로 깔끔하게 호출됩니다:

```cpp
FHitResult Hit;
GetWorld()->LineTraceSingleByChannel(Hit, Start, End, ECC_Visibility);

if (Hit.GetActor() && Hit.GetActor()->Implements<UDamageable>())
{
    IDamageable::Execute_TakeHit(Hit.GetActor(), Damage, AttackPower);
}
```

`Implements<>` 체크 + `Execute_TakeHit` 정적 호출 형태라 블루프린트 구현체에도 안전하게 동작합니다. 새 피격 가능 액터를 추가해도 발사 로직은 한 줄도 안 바뀌고, 캐스팅 분기가 사라지는 게 가장 큰 이점이에요.

### 4. "UMG를 C++로만 짠다는 게 구체적으로 어떤 코드인가요?"

`UUserWidget`을 상속한 위젯에서 `NativeConstruct()`를 오버라이드해 `WidgetTree`를 직접 조립합니다.

```cpp
void UNeonHUDWidget::NativeConstruct()
{
    Super::NativeConstruct();

    auto* Root = WidgetTree->ConstructWidget<UCanvasPanel>();
    WidgetTree->RootWidget = Root;

    ResourceText = WidgetTree->ConstructWidget<UTextBlock>();
    Root->AddChild(ResourceText);

    auto* Slot = Cast<UCanvasPanelSlot>(ResourceText->Slot);
    Slot->SetAnchors(FAnchors(0.f));
    Slot->SetPosition({40.f, 30.f});
    Slot->SetSize({200.f, 40.f});
}
```

자원 카운트·웨이브 라벨·HP 바·상점 메뉴 모두 이 방식. **위젯 블루프린트 에셋 0개**라서 UI 변경이 PR 디프에 그대로 잡힙니다.

단점은 시각적 미리보기 없이 좌표를 다뤄야 한다는 점이고, 디자인 변경이 잦은 상용 프로젝트라면 위젯 블루프린트 쪽이 맞습니다. 이번엔 "코드 100%" 컨셉을 위한 의도된 트레이드오프였어요.

### 5. "이중 크로스헤어는 화면에 정확히 어떻게 그리시나요?"

`ANeonHUD::DrawHUD()`에서 Canvas API로 매 프레임 두 좌표를 따로 계산해 그립니다.

```cpp
void ANeonHUD::DrawHUD()
{
    Super::DrawHUD();
    if (!bTurretBoarded) return;

    // 1) 카메라 중심 — 화면 정중앙
    const float CX = Canvas->ClipX * 0.5f, CY = Canvas->ClipY * 0.5f;
    DrawLine(CX-20, CY, CX+20, CY, FLinearColor::White, 2.f);
    DrawLine(CX, CY-20, CX, CY+20, FLinearColor::White, 2.f);

    // 2) 포신 방향 — 8000uu 전방 월드 위치를 스크린 투영
    const FVector BarrelEnd = Turret->Barrel->GetComponentLocation()
                            + Turret->Barrel->GetForwardVector() * 8000.f;
    FVector2D Screen;
    if (PlayerOwner->ProjectWorldLocationToScreen(BarrelEnd, Screen))
        DrawRect(FLinearColor::Yellow, Screen.X-3, Screen.Y-3, 6, 6);
}
```

포신은 `RotationSpeed`(deg/s) 제한으로 카메라 회전을 즉시 따라가지 못합니다. 이 둘의 차이가 그대로 두 조준점 간격이 되고, **그게 조준 지연의 시각 피드백이자 무게감의 정체**예요. `ProjectWorldLocationToScreen`이 화면 밖 좌표는 `false`를 반환해서, 시야 밖일 때는 점이 안 그려지는 부수 효과도 자연스럽게 처리됩니다.

### 6. "Dynamic Material Instance는 어떻게 만들고 제어하나요?"

부모 머티리얼은 `M_NeonBase` 하나만 에디터에서 만듭니다 — Vector 파라미터 `BaseColor`, Scalar 파라미터 `Glow`가 Emissive로 연결된 단순한 Unlit이고요.

런타임에는 `CreateDynamicMaterialInstance`로 MID를 만들고 파라미터를 코드로 제어합니다.

```cpp
void ABase::BeginPlay()
{
    Super::BeginPlay();
    MID = Mesh->CreateDynamicMaterialInstance(0, ParentMaterial);
    MID->SetVectorParameterValue(TEXT("BaseColor"), FLinearColor::Green);
    MID->SetScalarParameterValue(TEXT("Glow"), 3.f);
}

void ABase::Tick(float Dt)
{
    HitFlash = FMath::Max(0.f, HitFlash - Dt * 3.f);
    const FLinearColor C = FMath::Lerp(BaseColor, FLinearColor::Red, HitFlash);
    MID->SetVectorParameterValue(TEXT("BaseColor"), C);
}
```

머티리얼 에셋은 단 1개. 색·발광·피격 플래시·HP 25% 미만 경고색까지 같은 MID의 파라미터로 코드에서 제어합니다. 새 색상을 추가할 때 머티리얼 그래프를 안 건드려도 된다는 게 가장 큰 이점이에요.

### 7. "메모리 관리는 어떻게 하시나요? `UPROPERTY` 와 가비지 컬렉션은?"

UE의 GC는 `UPROPERTY()` 매크로로 마킹된 참조만 추적합니다. 일반 raw 포인터는 GC가 인식 못 해서, 가리키는 UObject가 회수되면 댕글링이 됩니다.

```cpp
// ✅ 강한 참조 — 가장 일반적
UPROPERTY()
UStaticMeshComponent* Mesh = nullptr;

// ✅ 약한 참조 — 순환 끊기 / 살아있는지만 확인할 때
UPROPERTY()
TWeakObjectPtr<APlayerShip> HomingTarget;

// ❌ raw 포인터 — GC 추적 안 됨
AMonster* CachedTarget;
```

NEON DRIFT에서 `TWeakObjectPtr`를 쓴 예가 `AResourceShard::HomingTarget`입니다. 자석 호밍 도중에 플레이어가 죽거나 파괴되면 강한 참조는 댕글링 위험이 있는데, weak로 잡으면 매 틱 `IsValid()` 한 줄로 안전하게 처리됩니다.

UObject가 아닌 데이터(자체 struct, 외부 라이브러리 객체)는 `TSharedPtr`/`TUniquePtr`로 갑니다.

### 8. "상태머신을 enum + switch로 짜셨는데, State 패턴 클래스 기반이 더 낫지 않나요?"

규모 트레이드오프입니다.

**enum + switch (지금 방식)**
- 상태가 단일 변수에 들어가서 디버거에서 한눈에 보임
- 신규 상태 추가가 enum에 한 줄 + switch에 한 case면 끝
- 전이 로직이 한 파일에 모여 있어 흐름 추적이 쉬움

**State 패턴 (클래스 기반)으로 가야 할 신호**
- 상태별 동작이 수백 줄 단위로 커지면서 GameMode가 비대해질 때
- 전이 조건이 복잡해질 때 (다중 조건, 우선순위, 인터럽트 등)
- state-local 변수가 많아져서 GameMode 멤버가 폭증할 때

NEON DRIFT는 GameMode 한 클래스로 충분한 규모였고, 클래스 분리는 **YAGNI** 라고 판단했습니다. 확장이 필요해지면 UE 5.x의 `StateTreeComponent`나 BehaviorTree로 자연스럽게 이관 가능합니다.

### 9. "빌드 의존성은 어떻게 관리하셨나요? `Build.cs` 는 어떻게 생겼나요?"

`NEONDRIFT.Build.cs`의 핵심부:

```csharp
PublicDependencyModuleNames.AddRange(new[] {
    "Core", "CoreUObject", "Engine", "InputCore",
    "EnhancedInput",   // 입력
    "UMG",             // 위젯
    "Niagara",         // 파티클
    "AudioMixer"       // 오디오 제어
});

PublicIncludePaths.AddRange(new[] {
    Path.Combine(ModuleDirectory, "Public/Actors"),
    Path.Combine(ModuleDirectory, "Public/Items"),
    Path.Combine(ModuleDirectory, "Public/Enemies"),
    // ...
});
```

`PublicDependencyModuleNames`에 모듈을 등록해야 그 모듈의 헤더를 `#include`로 끌어올 수 있고, `PublicIncludePaths`에 카테고리 폴더를 모두 추가해서 다른 파일에서 `#include "PlayerShip.h"`처럼 flat하게 쓸 수 있도록 했습니다.

추가 모듈은 Niagara 빼면 엔진 기본 의존성이라 컴파일 시간이 가벼운 편이에요.
